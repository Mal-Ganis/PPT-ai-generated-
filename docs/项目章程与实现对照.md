## 3. MOV 与质量指标对照

章程 **可衡量组织价值（MOV）** 与 **质量要求**：

| 指标 | 章程目标 | 当前实现 | 差距说明 |
|------|----------|----------|----------|
| 效率 | 中等复杂度主题全流程 **≤180 秒** | ❌ 未系统化测量 | 无耗时统计/达标报告；验收脚本幻灯片生成超时可达 **1900 秒** |
| 内容质量 | 人工平均 **≥4.2/5** | ⚠️ 机制有、量表未对齐 | 前端为 **0–100 分** 滑条，未映射 5 分制，缺批量统计 |
| 事实可靠性 | 与权威来源一致 **≥92%** | ⚠️ 有算法、需实测 | `factVerificationRate` + 自动分映射（≥0.92 满分）；无 SiliconFlow 时降级，难稳定达标 |
| 信息密度 | 每页 **≥3.5 条** 要点 | ⚠️ 部分 | Prompt 要求每页至少 3 条；无「3.5 条/页」硬性统计 |
| 大纲逻辑 | 人工评分 **≥88%** | ⚠️ 部分 | 自动结构分为启发式，非章程所述人工 88% 流程 |
| 工程稳定性 | 核心流程无崩溃、完成度 **100%** | ⚠️ 依赖环境 | JDK/数据库/API Key 未配齐会导致失败 |
| 语言表达 | 符合中文学术/商务规范 | ⚠️ 依赖模型 | 通过 DeepSeek Prompt 约束，无独立评测流水线 |

**自动评估与章程的代码对齐点**：`AutoEvaluationScoringService` 将 `factVerificationRate ≥ 0.92` 映射为事实维度满分，与章程 92% 目标一致。

## 8. 改进建议（按优先级）

### 高优先级（答辩 / 验收）

1. **交付物澄清或补做**：实现简易 PPTX 导出，或在章程/POS 中书面将交付物定为 **Markdown 草稿**。  
2. **MOV 实测表**：3～5 个中等主题，记录「主题 → 大纲 → 全文」耗时，说明是否满足 180 秒及优化措施。  
3. **评估对齐章程**：人工表改为 1～5 分（或换算），附平均分 ≥4.2、事实率 ≥92% 的截图/表格。  
4. **跑通验收**：`docker compose up -d` → 配置 Key → `mvn spring-boot:run` → `npm run dev` → `scripts/acceptance-test.ps1`。

### 中优先级（工程质量）

5. 统一 **JDK 21** 或将 `pom.xml` 与 README 均改为 **17**。  
6. 无 `SILICONFLOW_API_KEY` 时在前端/后端**明确提示**事实率为启发式。  
7. 更新 POS/章程附录：技术栈改为 **React + Spring Boot + PostgreSQL/pgvector + DeepSeek**。

### 低优先级（加分）

8. Markdown 导出增强引用块、配图占位。  
9. 项目列表展示最近评估总分 / 事实率。  
10. 新增 `SRS.md`，将 FR-01～FR-13 映射到现有 API。
